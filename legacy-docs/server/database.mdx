---
title: Database
description: SQLite database schema and how game stats are persisted.
---

The server uses SQLite (via JDBC) to persist player stats and game history. `DatabaseManager` (`flaggi.server.common`) handles all database operations.

## Schema

### `players`

Tracks lifetime stats per player.

| Column         | Type        | Description       |
| -------------- | ----------- | ----------------- |
| `id`           | INTEGER PK  | Auto-increment ID |
| `name`         | TEXT UNIQUE | Player username   |
| `total_kills`  | INTEGER     | Lifetime kills    |
| `total_deaths` | INTEGER     | Lifetime deaths   |
| `total_wins`   | INTEGER     | Games won         |
| `total_losses` | INTEGER     | Games lost        |
| `games_played` | INTEGER     | Total games       |
| `first_seen`   | TIMESTAMP   | First connection  |
| `last_seen`    | TIMESTAMP   | Last connection   |

### `games`

Records each completed game.

| Column          | Type       | Description              |
| --------------- | ---------- | ------------------------ |
| `id`            | INTEGER PK | Auto-increment ID        |
| `winner_team`   | TEXT       | "RED" or "BLUE"          |
| `duration_secs` | INTEGER    | Game duration            |
| `played_at`     | TIMESTAMP  | When the game was played |

### `player_game_stats`

Per-player breakdown for each game.

| Column      | Type       | Description                    |
| ----------- | ---------- | ------------------------------ |
| `id`        | INTEGER PK | Auto-increment ID              |
| `player_id` | INTEGER FK | References `players.id`        |
| `game_id`   | INTEGER FK | References `games.id`          |
| `team`      | TEXT       | "RED" or "BLUE"                |
| `kills`     | INTEGER    | Kills in this game             |
| `deaths`    | INTEGER    | Deaths in this game            |
| `won`       | BOOLEAN    | Whether this player's team won |

## Operations

### `saveGame(winnerTeam, durationSecs, playerStats)`

Called when a game ends. Wrapped in a transaction:

1. Insert or find each player (UPSERT pattern: `INSERT OR IGNORE` + `SELECT`)
2. Update lifetime stats (`total_kills`, `total_deaths`, `total_wins`/`total_losses`, `games_played`, `last_seen`)
3. Insert game record
4. Insert per-player game stats
5. Commit (or rollback on failure)

### `getLeaderboard()`

Returns all players sorted by:

1. Wins (descending)
2. K/D ratio (descending) - with `CASE` to handle division by zero

### `getRecentGames(limit)`

Returns the last N games with their player stats joined in.

## Database location

The SQLite database file is created in the server's working directory (typically a temp directory when launched via `run.sh`). Using the `-r` / `--rebuild` flag with `run.sh server` clears this directory, resetting the database.
