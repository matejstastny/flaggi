---
title: Game Renderer
description: How GameUi renders the game world, camera, and sprites.
---

`GameUi` (`flaggi.client.ui`) is the main game renderer. It extends `Renderable` and handles camera transforms, floor tiling, sprite rendering, and the HUD.

## Camera system

The camera keeps the local player centered on screen. Every frame:

1. **Translate** — shift the world so the local player is at screen center (`px(50), px(50)`)
2. **Zoom** — scale by `ZOOM = 0.5` around the player's screen position, effectively zooming out to show more of the map

```java
// Simplified camera transform
g2.translate(screenCenterX, screenCenterY);
g2.scale(ZOOM, ZOOM);
g2.translate(-playerWorldX, -playerWorldY);
```

## Floor rendering

The floor is drawn as a repeating tile grid. The `floor-tile` sprite is tiled across the entire room dimensions. Only tiles visible within the camera viewport are rendered (culled by screen bounds).

## Object rendering

Each `ServerGameObject` in the update is rendered based on its type:

| Type | Sprite | Behavior |
|------|--------|----------|
| `TREE` | `tree/tree.png` | Static, rendered at world position |
| `BULLET` | `bullet/bullet.png` | Static, rendered at world position |
| `FLAG` | `flag-blue/` or `flag-red/` | Selected by skin field |
| `PLAYER` | `player-{skin}/` | Animated, directional, can flip |

## Player rendering

Player sprites are cached in a `HashMap<String, Sprite>` keyed by username. On each frame:

1. Look up or create the `Sprite` for this player's skin
2. Set the animation based on `PlayerAnimation` (`idle`, `walk-up`, `walk-down`, `walk-side`, `walk-diagonal`)
3. If `facingLeft` is true, flip horizontally using `g2.scale(-1, 1)`
4. Call `sprite.tick()` and `sprite.render()`

### Animation mapping

| Proto enum | Sprite folder |
|------------|---------------|
| `ANIM_IDLE` | `idle` |
| `ANIM_WALK_UP` | `walk-up` |
| `ANIM_WALK_DOWN` | `walk-down` |
| `ANIM_WALK_SIDE` | `walk-side` |
| `ANIM_WALK_DIAGONAL` | `walk-diagonal` |

### Skin mapping

| Proto enum | Sprite folder |
|------------|---------------|
| `SKIN_BLUE` | `player-blue` |
| `SKIN_RED` | `player-red` |
| `SKIN_JESTER` | `player-jester` |
| `SKIN_VENOM` | `player-venom` |

## Client-side movement prediction

To keep movement feeling responsive, `GameUi` predicts the local player's position each frame using the same movement logic as the server:

1. Read held keys
2. Compute movement vector (speed = `PLAYER_SPEED` = 3.0 px/tick)
3. Attempt diagonal move with `tryMove` (AABB collision check against trees)
4. If blocked, try horizontal-only, then vertical-only (sliding collision)
5. Clamp to room boundaries

When the server's authoritative position arrives, it's compared to the predicted position. If the difference exceeds a snap threshold (60² pixels squared), the client snaps directly to the server position instead of smoothly correcting.

## Hitbox constants

The client mirrors the server's hitbox definitions for accurate collision prediction:

| Object | Offset X | Offset Y | Width | Height |
|--------|----------|----------|-------|--------|
| Player | `PLAYER_HB_OX` | `PLAYER_HB_OY` | `PLAYER_HB_W` | `PLAYER_HB_H` |
| Tree | `TREE_HB_OX` | `TREE_HB_OY` | `TREE_HB_W` | `TREE_HB_H` |
